# CGOS-SOLOCREW-RUNTIME-BLOCKER-INTAKE-v0.1

## Document Control

- Document ID: `CGOS-SOLOCREW-RUNTIME-BLOCKER-INTAKE-v0.1`
- Status: `Runtime-Family Intake Draft`
- Authority: `Mother-runtime research draft derived from sealed SoloCrew structural inputs`
- Scope: `SoloCrew-driven upstream runtime blockers only`

## Purpose

This note records only the runtime blockers that SoloCrew has now exposed after freezing its post-`v0.1` structural constitution.

It does not reopen broad mother-runtime redesign.
It does not treat SoloCrew product structure as automatic runtime law.

## Source Inputs

This intake is derived from:

- sealed SoloCrew `v0.1` release and closure documents
- SoloCrew structural constitution documents for:
  - `Cell`
  - `CEO Orchestrator / Crew Compiler`
  - management interface boundary
  - single-cell initialization
  - architecture planes
  - version roadmap
- current `Cognitive_OS` runtime, registry, binding, and projection handoff baselines

## Intake Rule

A SoloCrew demand is recorded here as an upstream runtime blocker only when all of the following are true:

1. the demand can no longer stay truthful as product-only structure
2. the demand requires reusable mother-runtime behavior or object law
3. the demand is not merely a product UI, DTO, or workflow packaging need
4. the demand is not yet a protocol promotion question

## Immediate Blockers

### Current judgment

There are `no immediate blockers` for the current docs-only wave.

Why:

- SoloCrew has only frozen structural constitutions so far
- those structures can still remain product-side for the current wave
- `Cognitive_OS` does not need an emergency runtime patch to keep SoloCrew truthful today

## Near-Term Blockers

These are the bounded runtime-family gaps most likely to matter once SoloCrew moves from structural docs into bounded post-`v0.1` implementation.

### 1. Cell / Portfolio runtime-family scoping

Current state:

- `project`, `agent-group`, `objective`, `review-cycle`, `memory-profile`, and `preference-profile` already exist
- there is no runtime-family object or scoping draft yet for `Cell` or `Portfolio`

Why this becomes a blocker later:

- SoloCrew now treats a single `Cell` as its first valid operating unit
- later management and portfolio expansion cannot stay coherent forever without a reusable scoping concept above `agent-group`

Bounded need:

- a runtime-family draft for `Cell`
- a runtime-family draft for `Portfolio`
- clear relation to existing `project` and workforce objects

### 2. Crew Compiler runtime surface

Current state:

- `runtime/core/runtime-orchestrator.ts` exposes bounded orchestration glue
- there is no runtime-family compile surface for translating organization state into crew topology and execution posture

Why this becomes a blocker later:

- SoloCrew has frozen `CEO Orchestrator` as a `Crew Compiler`
- once compile outputs need reusable runtime meaning rather than product-only planning meaning, `Cognitive_OS` needs a bounded runtime surface for that step

Bounded need:

- define compile-phase runtime surface
- keep it separate from execution-phase runtime
- avoid pretending provider execution is part of the compiler

### 3. Summary projection runtime capability

Current state:

- current runtime can expose local truth and bounded execution outcomes
- there is no mother-runtime capability for `full local truth -> bounded upward summary truth`

Why this becomes a blocker later:

- SoloCrew now has explicit `Cell` versus management-plane layering
- top-level management and later Secretary surfaces will need lawful summary inputs that are more structured than raw store reads and less product-specific than UI cards

Bounded need:

- define summary projection as a runtime capability
- keep UI rendering and card composition downstream

### 4. Role / model / tool / constraint / stop-loss policy surface

Current state:

- `policy-rule` exists in registry and runtime types
- `policy-service.ts` currently implements only the minimal confirm-gate path for the existing execution baseline

Why this becomes a blocker later:

- SoloCrew structural demands now include role policy bindings, capability supply, constraint sets, and escalation posture
- those demands cannot stay truthful forever if runtime has no bounded policy surface to interpret them

Bounded need:

- role-model binding policy
- tool policy
- cost / constraint policy
- stop-loss / escalation policy

### 5. Business / metrics pack mounting points

Current state:

- SoloCrew has frozen `Business Pack Mount` and `Metrics Pack Mount` as optional structural mounts
- `Cognitive_OS` has no draft mounting concept yet

Why this becomes a blocker later:

- if packs remain purely product-side forever, they cannot become lawful reusable runtime contributions
- if they move upward carelessly, they will contaminate runtime with product-specific dashboards and domain behavior

Bounded need:

- define runtime mounting points only
- keep domain dashboards and product packaging out of `Cognitive_OS`

## Not-Yet Blockers

These are not runtime blockers yet.
They should not drive current mother-runtime changes.

- `Cell Policy Profile`
- `Management Directive`

- generic escalation / approval envelope semantics

Reason:

- SoloCrew has not yet proven these belong in protocol constitution
- `Cognitive_OS` should not promote them upward by convenience

### 3. Broad runtime expansions outside the bounded SoloCrew ask

Still not the current concern:

- provider bridge implementation
- channel runtime
- budget runtime
- full PSG
- full VSL

Reason:

- SoloCrew structural docs did not reopen these as immediate requirements
- the sealed `v0.1` boundary still holds

## Net Intake Judgment

Current wave judgment:

- no immediate mother-runtime blocker needs code
- near-term runtime-family drafts are warranted now
- protocol promotion is not warranted now
- the correct response is a bounded research pack, not a redesign wave
