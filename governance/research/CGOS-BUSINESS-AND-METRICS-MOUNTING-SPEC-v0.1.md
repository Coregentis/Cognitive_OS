- Phase Constraint: `Docs-only. No pack runtime, dashboard, or marketplace implementation is created by this note.`

## Purpose

This note defines the bounded mounting concept for:

- `Business Pack Mount`
- `Metrics Pack Mount`

The trigger is SoloCrew's structural choice to treat both as optional mounts rather than core day-one constitutional objects.

## Mounting Principle

If these mounts later move into `Cognitive_OS`, the mother-runtime should provide:

- lawful mounting points
- neutral attach semantics
- bounded policy and summary integration hooks

It should not provide by default:

- domain dashboards
- product KPI pages
- business-specific UI
- full domain operating systems

## Business Pack Mount

### Runtime meaning

`Business Pack Mount` is the bounded runtime attach point for domain operating contributions such as:

- role or crew-shape hints
- objective / work templates
- constraint overlays
- tool-policy overlays
- bounded compile or review inputs

### What it should not become

It should not become:

- a domain dashboard
- a product package manager UI
- a protocol concept by default
- a back door for business-specific runtime law to bypass governance

## Metrics Pack Mount

### Runtime meaning

`Metrics Pack Mount` is the bounded runtime attach point for metrics-related contributions such as:

- metric definitions
- evaluation thresholds
- summary projection hints
- review and escalation measurement inputs

### What it should not become

It should not become:

- a KPI cockpit
- a reporting UI
- an analytics warehouse
- a budget runtime by another name

## Relation To Cell / Portfolio

The likely bounded placement is:

- packs may mount at `Cell` scope first
- later they may mount at `Portfolio` scope where that is lawful and useful

The mount model should therefore be scoping-aware without forcing portfolio-first initialization.

## Relation To Policy / Summary Projection

Mounts may later contribute to:

- runtime policy inputs
- crew compilation inputs
- summary projection hints

But they should not replace:

- the generic runtime policy surface
- the generic summary projection surface

## Non-Goals

This draft does not define:

- a pack marketplace
- business-pack schemas
- metrics-pack schemas
- domain dashboards
- pack UI
- protocol promotion

## Net Draft Judgment

The correct mother-runtime role is narrow:

- provide mounting points
- provide governance hooks
- provide scoping hooks

The correct downstream product role is broader:

- package domain behavior
- present dashboards
- render metrics surfaces
