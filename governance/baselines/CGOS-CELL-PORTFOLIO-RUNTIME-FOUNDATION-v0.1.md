# CGOS-CELL-PORTFOLIO-RUNTIME-FOUNDATION-v0.1

## Purpose

This baseline freezes the narrowest lawful runtime-facing interpretation of multi-cell existence that `Cognitive_OS` may later need in response to SoloCrew `v0.4`.

It does not define:

- Secretary behavior
- portfolio UI
- enterprise governance
- implemented runtime code

## Current Repo Truth To Reuse

Current `main` already provides neutral scoping and workforce surfaces such as:

- `project`
- `agent-group`
- `agent-worker`
- `objective`
- `work-item`
- `review-cycle`
- `memory-profile`
- `preference-profile`
- workforce lifecycle and state persistence

Those surfaces remain the lawful current truth.

This baseline only freezes what a later runtime-family extension would need to mean if current scoping becomes insufficient.

## Narrow Runtime-Facing Interpretation

### `Cell`

If later promoted, `Cell` should be interpreted as:

- a bounded runtime-family operating scope
- narrower than a broad portfolio
- wider than a single worker or work-item
- capable of lawfully referencing one coherent business/work unit inside a project scope

The minimum runtime-facing implications are:

- identity
- scoping boundary
- lawful references to relevant local workforce/runtime truth
- continuity and evidence boundary

### `Portfolio`

If later promoted, `Portfolio` should be interpreted as:

- a bounded aggregation scope over one or more `Cell`s
- a later runtime-family layer than `Cell`
- a neutral coordination scope, not a product cockpit

The minimum runtime-facing implications are:

- identity
- membership boundary
- lawful aggregation boundary
- cross-cell governance context

## Minimum Foundation Elements

The minimum later runtime foundation would need to cover:

### 1. Identity

Any later `Cell` or `Portfolio` runtime family would need stable identity and scope references.

That identity must remain:

- neutral
- runtime-family safe
- separate from product card or UI identity

### 2. Scope Boundary

Any later `Cell` scope would need to answer:

- what local workforce/runtime truth it may govern or reference
- what sits inside versus outside its boundary
- how it relates to `project`

Any later `Portfolio` scope would need to answer:

- which `Cell`s participate
- what aggregation boundary is lawful
- what remains below portfolio scope

### 3. Evidence And Continuity Implications

If multi-cell runtime scoping later exists, it would need lawful treatment of:

- continuity boundaries
- evidence boundaries
- review or governance posture boundaries

This does not mean broad event-history or workflow timeline implementation.
It only means later runtime-family scoping cannot ignore evidence/governance implications entirely.

### 4. Governance And Policy Implications

Any later runtime-family scoping would likely interact with:

- policy posture
- escalation posture
- later summary projection inputs

It must still remain separate from:

- product UX
- Secretary behavior
- portfolio dispatch surfaces

## What This Baseline Does Not Claim

This baseline does not claim that `Cognitive_OS` already has:

- runtime `Cell` objects
- runtime `Portfolio` objects
- cross-cell orchestration
- cross-cell scheduling
- portfolio dispatch
- Secretary behavior

It only freezes the narrowest lawful direction if later blocker waves justify moving beyond current workforce scoping.

## Foundation Rule

If a proposed `Cell` or `Portfolio` runtime feature mainly serves:

- product cards
- dashboard layout
- Secretary UX
- enterprise tenancy

then it does not belong in this baseline by default.
