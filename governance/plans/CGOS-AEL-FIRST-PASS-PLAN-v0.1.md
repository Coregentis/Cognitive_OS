# CGOS-AEL-FIRST-PASS-PLAN-v0.1

## Purpose

This plan freezes what `AEL first pass` means in current `Cognitive_OS` repo
reality.

It exists because activation behavior must not be invented from downstream
product pressure, and must not be promoted into MPLP protocol law merely
because the runtime now carries activation-facing nouns.

The required order remains:

- MPLP truth source
- ratified Coregentis crosswalk / binding interpretation
- bounded runtime-private realization
- only later downstream projection

## Current Repo Reality

`Cognitive_OS` already has:

- a bounded executable minimal cognitive loop
- runtime-private activation objects:
  - `activation-signal`
  - `action-unit`
- bounded `PolicyService`, `ConfirmService`, and `TraceService`
- VSL first-pass continuity substrate
- PSG first-pass semantic relation substrate
- Delta Drift & Impact first pass over the requirement-change path

`Cognitive_OS` does not yet have:

- provider execution
- dispatch execution
- full AEL realization
- a full policy engine
- a rich confirm workflow
- downstream SoloCrew exception-plane projection

That means the lawful next move is a runtime-private governed activation
assessment layer, not execution-stack expansion.

## What AEL First Pass Means Now

In current repo reality, `AEL first pass` means:

- activation scope is made explicit for the bounded minimal runtime path
- action-unit lifecycle is evaluated through a runtime-private AEL assessment
- policy gating is represented rather than assumed
- confirm-gate linkage is preserved when confirm is required
- suppression metadata is explicit when policy blocks activation
- escalation metadata is explicit when reconcile tension means the action should
  not continue as a normal active path
- trace / decision linkage may be attached to the runtime-private AEL
  assessment

This wave remains bounded to:

- current project-local runtime context only
- one deterministic minimal-loop action-unit path
- runtime-private assessment only
- no provider execution
- no dispatch execution
- no product review queue or founder packet law

## What This Wave Includes

- explicit activation scope reading from runtime context
- runtime-private AEL assessment outcome:
  - `activate`
  - `confirm_required`
  - `suppressed`
  - `escalate`
- action-unit lifecycle interpretation tied to that assessment
- policy-rule linkage through matched rule identifiers
- confirm-gate linkage when applicable
- suppression metadata
- escalation metadata
- bounded trace/evidence linkage when trace and decision records exist
- attachment of the assessment to runtime run output without widening protocol
  export scope

## What This Wave Explicitly Excludes

- provider/channel execution
- direct dispatch / execute semantics
- retry policies
- sandbox/provider adapters
- full policy-engine richness
- full confirm workflow richness
- product review queue semantics
- founder packet semantics
- SoloCrew DTOs
- MPLP top-level object promotion
- protocol export widening

## Why AEL Remains Runtime-Private

Current MPLP truth gives:

- informative runtime guidance for `AEL`
- protocol-facing execution / trace / confirm evidence families

Current MPLP truth does not give:

- a top-level protocol-native `AEL` object
- a mandated execution engine
- product-specific workflow semantics

Therefore this wave may enrich runtime-private activation assessment inside
`Cognitive_OS`, but it may not claim new MPLP object law or downstream product
workflow law.

## Crosswalk Gate Compliance

This wave follows the ratified crosswalk by preserving:

- `AEL` as a Coregentis-private runtime substrate
- `activation-signal` and `action-unit` as runtime-private objects
- `confirm-gate` as runtime-bound rather than newly invented protocol law
- trace / decision evidence as the only protocol-adjacent output surface

The gate is satisfied only if this implementation:

- reuses MPLP truth without widening it
- reuses current Coregentis registry and binding truth
- keeps AEL assessment runtime-private
- does not export AEL internals as MPLP-native artifacts
- does not project directly into SoloCrew

## Relation To Existing Runtime Services

This wave should sit on top of the current bounded services:

- `ActivationService`
  - still creates `activation-signal` and `action-unit`
- `PolicyService`
  - still provides the first gating signal
- `ConfirmService`
  - still provides bounded confirm-gate linkage
- `TraceService`
  - still provides trace / decision evidence after the bounded activation path

`AEL first pass` does not replace those services.
It interprets them through a governed activation assessment boundary.

## Relation To VSL / PSG / Delta Drift

This wave may use existing runtime context from:

- VSL continuity state
- PSG semantic relation state
- Delta Drift / conflict tension already detected in the current bounded path

But this wave does not:

- implement new drift logic
- implement graph propagation logic
- implement learning logic

It only consumes already-available runtime context when that context affects
whether an action should continue, be suppressed, or be escalated.

## Relation To Later SoloCrew Projection

Later SoloCrew work may consume projected activation outcomes, exceptions, or
review surfaces.

This wave does not create those projection contracts.

It only prepares them by making runtime-private activation assessment explicit
inside `Cognitive_OS`.

## Boundary Conclusion

`AEL first pass` is complete only if:

- activation scope is explicit
- policy gating is explicit
- confirm linkage is explicit when required
- suppression and escalation metadata are explicit
- action-unit lifecycle remains deterministic
- no provider execution or product workflow law is claimed
- protocol export scope remains unchanged
