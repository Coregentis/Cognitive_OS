# CGOS Workflow-Truth Candidate Rules v0.1

## Purpose

This backlog rule document records what future candidate discipline would be
required before any workflow-truth family for handoff/review/revision semantics
could lawfully enter `Cognitive_OS`.

This is governance only.
It does not create or imply an adopted runtime family.

## Candidate Rules

Any future workflow-truth candidate must satisfy all of the following rules.

### Rule 1: Repeated Downstream Usage Alone Is Insufficient

Repeated downstream use in SoloCrew is not enough by itself.

Why:

- downstream repetition can show pressure
- downstream repetition does not by itself prove neutral upstream kernel

### Rule 2: Product Convenience Alone Is Insufficient

Convenience for product organization, UI wording, or shelf consistency is not
enough by itself.

Why:

- mother-runtime truth must be smaller and more neutral than product surface
  convenience

### Rule 3: Candidate Meaning Must Reduce To Neutral Runtime-Private Truth

Any future candidate must reduce to neutral runtime-private truth rather than
SoloCrew-specific page or packet semantics.

That means the candidate must avoid:

- Secretary-first wording
- shelf-first wording
- page-routing language
- review-packet UI vocabulary unless that vocabulary can be reduced further

### Rule 4: Candidate Must Survive SoloCrew Downstream Check

Any future candidate must be checked against current SoloCrew usage and must
improve clarity there rather than merely mirroring current product language.

The candidate fails this rule if it:

- simply copies current product packet states upward
- makes downstream boundaries blurrier
- turns current product posture into implied execution authority

### Rule 5: Candidate Must Survive MPLP Non-Promotion Check

Any future candidate must be checked against current MPLP non-promotion
boundaries.

The candidate fails this rule if it:

- sounds like protocol promotion by convenience
- creates pressure to read runtime-private objects as protocol objects
- makes downstream packet-state adoption sound like MPLP law

### Rule 6: Candidate Must Survive Machine-Readable Schema Discipline

Any future candidate must be expressible as a coherent machine-readable family.

That requires:

- schema discipline
- explicit authority class
- explicit required and optional fields
- explicit object-local status meaning
- explicit non-goal language

### Rule 7: Candidate Must Survive Registry / Binding / Export Discipline

Any future candidate must be classifiable cleanly in:

- registry notes
- binding matrix
- export rules
- runtime typing
- frozen-truth loading

The candidate fails this rule if classification would be hand-wavy or if it
would accidentally imply protocol exportability.

### Rule 8: Candidate Must Not Smuggle In Workflow Authority

Any future candidate must remain below:

- direct approve control
- direct reject control
- direct dispatch control
- direct execute control
- workflow-engine ownership

If the candidate cannot stay clearly below those meanings, it is not lawful to
introduce as a bounded runtime-private family.

## Net Candidate Rule

The correct current backlog reading is:

- future workflow-truth discussion is allowed only as explicit candidate work
- current downstream packet states are not already upstream candidates by
  default
- any future adoption must pass downstream, MPLP, schema, registry, binding,
  export, and authority-boundary discipline together
